## 总结

第 11 课给模型调用加了一层错误恢复。

以前这条链路是：

```text
请求 DeepSeek -> 成功就继续 -> 失败就退出
```

现在 HarnessX 会先读取失败信号，再决定下一步动作：

```text
① 输出被截断 -> 扩大输出上限，仍不够就接着写
② 上下文超限 -> 紧急压缩，再请求一次
③ HTTP 429 / 529 -> 等一会儿，再发同一个请求
```


```text
错误信号 -> 分类 -> 修改恢复状态 -> 有限重试 -> 成功或明确停止
```

`错误`进入 Harness 的控制流程，成为**下一步动作的输入**。

每条恢复路径都有次数上限。程序知道什么时候继续，也知道什么时候停下来。

## 重点

- DeepSeek 会通过不同位置告诉程序发生了什么：
	- HTTP 状态码告诉程序请求有没有被服务端接受。
	- HTTP `Retry-After` 告诉程序应该等多久。
	- `choices[0].finish_reason` 告诉程序回答为什么结束。
	- 错误正文里的上下文标记告诉程序输入是否太长。
- HarnessX 根据三类信号选择三条恢复路径：
	- `finish_reason=length`：
		- 扩大 `max_tokens`，然后按需续写。
	- 上下文超限：
		- 调用第 8 课的 reactive compact，再重试一次。
	- HTTP 429 / 529：
		- 指数退避，连续 529 时可以切换备用模型。
- 恢复动作发生在工具执行前：
	- 同一个模型请求可以安全重发。
	- 已经返回但不完整的工具调用不会执行。
	- API 重试不会消耗正常的 Agent turn。
- 恢复必须有上限：
	- 临时故障最多重试 10 次。
	- reactive compact 最多尝试 1 次。
	- 输出最多续写 3 次。

## 用夜宵店接外卖单理解错误恢复

把一次 Agent 任务想成**一家夜宵店接到一张外卖单**。

用户下单，HarnessX 像店长，DeepSeek 像后厨。店长把订单递给后厨，后厨开始出菜。中间可能遇到三类问题，每类问题都能从现场信号判断出来。

- 小票打印到一半断了：
	- 这对应模型返回 `finish_reason=length` 或 `max_tokens`。
	- 第一次先换成长一点的小票纸，让后厨按原订单重新打一遍。
	- 长纸仍然断，就把已经打出来的内容夹好，再补一句“从断掉的地方继续打”。
	- 续写最多三次，避免一张订单无限拖下去。
- 操作台堆满了旧小票和食材：
	- 这对应上下文超限。
	- 店长把早期材料整理成一张摘要，贴到旁边。
	- 操作台上只保留摘要和最近还在用的材料。
	- 腾出位置后，把当前订单继续递给后厨。
- 平台爆单，取餐窗口暂时排队：
	- 这对应 HTTP 429 或 529。
	- 店长不会立刻反复催单，而是先等 0.5 秒，再等 1 秒、2 秒、4 秒。
	- 每次多错开一点随机时间，避免所有订单同时重新冲向后厨。
	- 连续三次遇到 529 时，如果配置了备用模型，就换一个后厨继续做同一张订单。

这三个现场信号对应三种恢复动作。

## 错误恢复夹在模型请求和 Agent Loop 之间

第 11 课没有增加新工具。

它在原来的模型请求外面加了一个恢复控制层。

![243. 技术：HarnessX 第 11 课：让 Agent 识别三类模型故障，用重试、压缩和续写恢复执行 图表 2](/assets/os/mermaid-dca82f975eeab7987941f19a4f4c1589498c0a5677f17a170012acf121cc2d40.svg)

这里要分清两个职责。

- `src/deepseek.js` 保存事实：
	- HTTP status 是多少。
	- 响应正文是什么。
	- `Retry-After` 是多久。
	- `finish_reason` 是什么。
- `src/agent-loop.js` 做决定：
	- 要不要重试。
	- 等多久。
	- 要不要压缩上下文。
	- 要不要扩大输出上限。
	- 什么时候停止。

传输层保留真实信号，Harness 层根据任务状态恢复。

## 三条恢复路径

### 输出被截断：先扩大上限，再续写

DeepSeek 的正常响应里，`message` 和 `finish_reason` 位于不同位置。

```json
{
  "choices": [
    {
      "message": {
        "role": "assistant",
        "content": "已经生成的回答"
      },
      "finish_reason": "length"
    }
  ]
}
```

`length` 表示回答碰到了本次 `max_tokens` 上限。

HarnessX 的恢复顺序是：

![243. 技术：HarnessX 第 11 课：让 Agent 识别三类模型故障，用重试、压缩和续写恢复执行 图表 3](/assets/os/mermaid-63a1cb6a3b97d64b273720793bb88c3c0f8830594a432c5e946c1ffda4bceacc.svg)

第一次扩容时，截断文本不会写进 `messages`。

原因很直接：当前请求还可以获得一份更完整的回答。保持原请求不变，能避免同一段内容被重复带入上下文。

扩大上限后仍然截断，HarnessX 才会：

- 保存当前 assistant 文本。
- 追加一条 user 续写提示。
- 让模型从中断位置继续。
- 合并相邻片段时去掉重复边界。

如果截断响应里带着不完整的 `tool_calls`，程序直接停止。半截工具参数不能交给本机执行。

### 上下文超限：紧急压缩一次

第 8 课已经在每轮请求前做四层上下文整理。

如果 DeepSeek 仍然返回 413，或者错误正文明确包含 `context_length_exceeded`、`prompt_too_long` 这类标记，HarnessX 会触发 reactive compact。

```text
写入完整 transcript
-> 用 DeepSeek 生成连续性摘要
-> 保留 system message、摘要和最近安全消息
-> 重新请求一次
```

这里有一个重要边界：

```text
普通 HTTP 400 不会自动触发压缩
```

400 可能代表参数错误、模型名错误或其他请求问题。把所有 400 都当成上下文太长，会掩盖真正的 bug。

reactive compact 只试一次。压缩后仍然超限，错误会继续抛出。

### HTTP 429 / 529：等一会儿再发

HTTP 429 表示请求太频繁，529 表示服务端过载。

HarnessX 保持当前 `messages` 不变，只重新发送模型请求。

等待时间使用指数退避：

```text
第 1 次：500ms + 0~25% 随机抖动
第 2 次：1000ms + 0~25% 随机抖动
第 3 次：2000ms + 0~25% 随机抖动
...
最多增长到 32000ms
```

如果服务端返回 `Retry-After`，直接采用服务端给出的等待时间。

连续三次 529 时：

- 配置了不同的 `DEEPSEEK_FALLBACK_MODEL`：
	- 切换到备用模型。
	- 保留当前任务和 messages。
	- 继续剩余重试。
- 没有配置备用模型：
	- 保持当前模型。
	- 继续按退避规则重试。

## 一次 HTTP 请求如何进入恢复流程

工具调用只会在恢复控制层返回一条完整模型消息后执行。

所以 429 / 529 重试不会重复执行工具。

## 怎么跑

沿用第 0 课的 `npm link` 和 DeepSeek 配置。

先跑正常工具循环：

```bash
hx agent "读取 README.md 前 40 行，只回答当前课程列表最后一项的编号和名称"
```

关键输出：

```bash
# 第一轮让 DeepSeek 决定读取文件
> read_file path=README.md limit=40

# 第二轮根据工具结果回答
当前课程列表最后一项是 `11`：Error Recovery 错误恢复。
```

再把本次进程的输出上限临时调小，真实触发输出截断：

```bash
DEEPSEEK_MAX_TOKENS=64 \
DEEPSEEK_ESCALATED_MAX_TOKENS=512 \
hx agent "用至少 500 字解释当前 HarnessX 的三条恢复路径：finish_reason=length、上下文超限、HTTP 429/529。只输出正文，不调用工具"
```

这条命令仍然调用真实 DeepSeek。两个环境变量只缩小本次请求的输出空间。

本次实际观察到：

```bash
# 64 token 不够，保持原 messages，把上限扩大到 512
[RECOVERY] output truncated max_tokens=64 -> 512

# 512 token 仍然不够，保存已有片段，再请求模型接着写
[RECOVERY] continuation=1/3

# 最终正常结束；恢复请求没有占用新的 Agent turn
[HOOK] Stop: turns=1, tool_calls=0, blocked=0
```

判断跑通的标准：

- 能看到 `output truncated`。
- 能看到一次 `continuation`。
- 最终能得到合并后的完整文本。
- `Stop` 仍然显示 `turns=1`。

429 / 529 依赖真实服务状态，不通过假错误制造。实际遇到时观察：

```bash
[RECOVERY] HTTP 429 retry=1/10 wait=0.6s model=deepseek-chat
[RECOVERY] HTTP 529 retry=3/10 wait=2.4s model=deepseek-chat
[RECOVERY] fallback model=备用模型名
```

## 这一课真正要记住

- Error signal：
	- 模型和 HTTP 服务返回的真实失败信息。
- Recovery state：
	- 当前模型、token 上限、续写次数、连续 529 次数。
- Retry budget：
	- 每条恢复路径允许消耗的最大次数。
- Exponential backoff：
	- 每次失败后逐步增加等待时间。
- Jitter：
	- 在等待时间上增加随机量，降低同时重试造成的新拥堵。
- Fallback model：
	- 主模型持续过载时可以切换的备用模型。

一句话收束：

```text
Harness 读取错误信号，用有限状态和有限次数把任务带回正常 Agent Loop。
```

### 代码细分

先为每次 `hx agent` 任务创建独立状态：

```js
function createRecoveryState() {
  return {
    // 输出截断时从默认上限提升到更大的上限。
    defaultMaxTokens: 4096,
    escalatedMaxTokens: 8192,
    maxTokens: 4096,

    // 控制扩容和续写次数。
    hasEscalated: false,
    continuationCount: 0,
    outputParts: [],

    // 控制 529 和备用模型。
    consecutive529: 0,
    currentModel: process.env.DEEPSEEK_MODEL || "deepseek-chat",
    fallbackModel: process.env.DEEPSEEK_FALLBACK_MODEL || "",
  };
}
```

HTTP 429 / 529 只包住模型请求：

```js
async function callWithTransientRetry(messages, options, state) {
  let retryCount = 0;

  while (true) {
    try {
      return await callDeepSeekMessage(messages, {
        ...options,
        model: state.currentModel,
        max_tokens: state.maxTokens,
      });
    } catch (error) {
      const status = transientStatus(error);

      // 只恢复 429 / 529，其他错误交回外层分类。
      if (!status || retryCount >= 10) {
        throw error;
      }

      retryCount += 1;
      const delayMs = retryDelayMs(retryCount - 1, error.retryAfterMs);
      await sleep(delayMs);
    }
  }
}
```

输出截断先扩容，第二次才保存片段：

```js
if (message.finish_reason === "length") {
  if (!state.hasEscalated) {
    // 第一次截断不改 messages，直接用更大上限重发原请求。
    state.hasEscalated = true;
    state.maxTokens = state.escalatedMaxTokens;
    continue;
  }

  if (message.tool_calls?.length) {
    // 半截工具参数不能执行。
    throw new Error("Incomplete tool call.");
  }

  // 保存已经生成的文本，让模型从中断位置继续。
  messages.push(normalizeAssistantMessage(message));
  state.outputParts.push(extractText(message.content));
  messages.push({ role: "user", content: CONTINUATION_PROMPT });
  continue;
}
```

上下文错误只恢复明确匹配的情况：

```js
if (
  isContextLengthError(error)
  && reactiveRetries < MAX_REACTIVE_COMPACT_RETRIES
) {
  reactiveRetries += 1;

  // 沿用第 8 课：保存 transcript、生成摘要、保留最近安全消息。
  await reactiveCompact(messages, stats, error);
  continue;
}

// 普通 400 和其他错误不会被误判成上下文过长。
throw error;
```
