---
title: "三个 AI 编程工具的配置边界：规则、命令、Skills 和 MCP 放哪里"
---

最近我把 832OS 里的 Agent 配置重新整理了一遍，最后发现一个问题：Codex、Cursor、Claude Code 看起来都支持规则、命令、Skills、MCP，但它们不是一套目录，也不应该强行塞进同一个目录。

我的结论先放前面：

```text
项目规则，留在项目里
项目命令，放到各工具自己的项目目录
用户级 Skills，单独维护到 ~/832/skills
MCP，不和 Skills 混放，按是否共享和是否含密钥决定位置
```

这不是为了整洁而整洁，而是为了以后不分叉、不污染项目，也不把私有配置误放进公开仓库。

## 1. 最终分工

我现在会按这几层来分：

| 类型         | 放哪里            | 为什么                         |
| ---------- | -------------- | --------------------------- |
| 工程规则       | 项目仓库           | 只对这个工程生效，应该跟代码和文档一起走        |
| 项目命令       | 各工具自己的项目目录     | slash command 是工具能力，不同工具不兼容 |
| 用户级 Skills | `~/832/skills` | 跨仓库复用，不属于某个工程               |
| MCP        | 工具自己的 MCP 配置   | MCP 是外部工具连接，不是知识文档          |
| 敏感配置       | 用户级或本地级        | 不进仓库，不跟项目共享                 |

一句话：**工程里只放工程需要的东西，个人能力放个人能力库，外部工具连接按权限边界管理。**

## 2. 三个工具怎么放

下面这张表，是我现在最想记住的版本。

| 能力         | Cursor Agent                              | Codex                                                 | Claude Code                                           |
| ---------- | ----------------------------------------- | ----------------------------------------------------- | ----------------------------------------------------- |
| 项目规则       | `.cursor/rules/*.mdc`，简单项目也可用 `AGENTS.md` | `AGENTS.md`                                           | `CLAUDE.md` 或 `.claude/CLAUDE.md`                     |
| 项目命令       | `.cursor/commands/*.md`                   | 通常用 repo skill 承接，比如 `.agents/skills/<name>/SKILL.md` | `.claude/commands/*.md`                               |
| 项目 Skills  | Cursor 没有同名原生 Skills 入口，通常转成 Rules / Commands | `.agents/skills/<name>/SKILL.md`                      | `.claude/skills/<name>/SKILL.md`                      |
| 用户级 Skills | 通常转成 User Rules、全局 Commands 或外部安装器适配       | `$HOME/.agents/skills`                                | `~/.claude/skills`                                    |
| MCP        | `.cursor/mcp.json` 和 `~/.cursor/mcp.json` | `~/.codex/config.toml`，可信项目也可用 `.codex/config.toml`   | `local / project / user` 三种 scope，项目共享配置是 `.mcp.json` |
| 适合放什么      | 项目规则、对话命令                                 | 项目说明、可复用工作流、外部工具配置                                    | 项目记忆、命令、复杂 Skills、MCP                                 |

这张表背后的重点是：**能共享的是思想和源头，不是目录名。**

比如都叫 Skills，但 Codex 看 `.agents/skills`，Claude Code 看 `.claude/skills` 和 `~/.claude/skills`。如果硬把三个工具指向同一个目录，短期看省事，长期一定会遇到兼容问题。

## 3. 规则：最适合放在工程里

规则是最适合进工程的。

因为规则描述的是这个项目本身：

- 这个项目是什么。
- 哪些目录能读，哪些默认不读。
- 公开内容和私有内容怎么区分。
- 发布链路怎么走。
- 哪些脚本不能擅自执行。

在 832OS 里，我现在把 `AGENTS.md` 当作通用工程契约。Codex 会读它，Cursor 也可以把它当作简单项目规则。

Claude Code 不直接读 `AGENTS.md`，所以我加了一个很薄的 `CLAUDE.md`：

```md
@AGENTS.md

## Claude Code

- 项目命令放在 `.claude/commands/`。
```

这样好处很明确：通用规则只维护一份，Claude Code 只补自己的差异。

## 4. 命令：不要追求跨工具共享

slash command 看起来都叫命令，但其实是每个工具自己的 UI 和触发机制。

所以项目命令我不再做统一目录，而是各放各的：

```text
.cursor/commands/new-issue.md
.claude/commands/new-issue.md
.agents/skills/new-issue/SKILL.md
```

这里文件名保留工具能识别的结构，内容保持同一个意图。

比如 832OS 里的 `/new-issues`，我希望它做的事很简单：

1. 后面什么都不写，就根据上下文生成 Issue。
2. 后面写一行，就按这一行生成 Issue。
3. 后面写多行，第一行是标题，后面是正文。

这里不需要搞成一个复杂 Skills，也不需要走 Raycast 剪贴板。它就是一个项目命令，目标很明确：把当前想法记到这个仓库的 Issue 里。

## 5. Skills：用户级的东西，不应该塞进项目

我之前容易混的地方就在这里。

像这些能力：

- 压缩 PNG
- 生成二维码
- 下载播客
- 翻译技术文档
- 写固定格式的总结

它们不是 832OS 这个工程独有的能力。今天在 832OS 里用，明天也可能在另一个仓库用。

所以它们不应该长期放在某个项目里，而应该维护到独立的用户级 Skills 工作区：

```text
~/832/skills
```

我的理解是：

```text
~/832/skills 负责维护源头
Codex / Cursor / Claude Code 只负责消费
```

这和项目里的 `.cursor/`、`.claude/`、`.agents/` 不冲突。

项目目录解决“这个工程怎么协作”，Skills 工作区解决“我这个人有哪些长期可复用能力”。

## 6. MCP：它不是 Skills，也不是规则

MCP 最容易被误放。

它不是提示词，也不是知识文档，而是外部工具连接。比如 GitHub、浏览器、Figma、文档服务、数据库、内部系统。

所以 MCP 的判断标准不是“它属于哪个 Agent”，而是：

```text
这个连接是否只给我自己用？
这个连接是否包含密钥？
这个连接是否必须让项目成员共享？
```

我的默认判断是：

| 场景 | 放法 |
|---|---|
| 个人常用工具，不需要共享 | 用户级配置 |
| 含 token、cookie、个人账号 | 用户级或本地级，不进仓库 |
| 项目成员都需要的非敏感工具 | 项目级配置 |
| 只是某个工程临时调试 | 本地项目级，不提交 |

对应到工具里：

- Codex 的 MCP 主要在 `~/.codex/config.toml`，可信项目也可以有 `.codex/config.toml`。
- Claude Code 可以按 `local / project / user` 配，项目共享配置是 `.mcp.json`。
- Cursor 的项目 MCP 在 `.cursor/mcp.json`，全局 MCP 在 `~/.cursor/mcp.json`，编辑器和 `cursor-agent` 可以共享配置。

所以 MCP 不应该跟着 Skills 一起搬。Skills 是能力说明，MCP 是工具连接。它们可以配合，但不是一类东西。

## 7. 832OS 现在怎么做

832OS 现在是一个工程，不再承担“用户级 Skills 仓库”的职责。

项目里保留这些：

```text
AGENTS.md
CLAUDE.md
.cursor/
.claude/
.agents/
automation/
```

它们分别负责：

- `AGENTS.md`：跨工具通用工程规则。
- `CLAUDE.md`：Claude Code 入口，导入 `AGENTS.md`。
- `.cursor/`：Cursor 的项目规则和命令。
- `.claude/`：Claude Code 的项目命令。
- `.agents/`：Codex 的项目 skill / command 适配。
- `automation/`：真实会执行的本机脚本。

而用户级 Skills，后续都进：

```text
~/832/skills
```

这样边界就清楚了。

## 8. 我的最佳实践

如果只记几条，我会这么记：

第一，工程规则优先写在工程里。

项目如何构建、如何发布、哪些目录不能乱看，这些都属于工程规则。它应该跟项目走，不应该藏在某个工具的用户配置里。

第二，命令按工具适配，不追求同目录共享。

Cursor、Codex、Claude Code 对 slash command 的支持方式不同。共用同一个文件夹不是目标，共用同一个意图才是目标。

第三，Skills 要有独立源头。

只要一个能力会跨项目使用，就不要放进某个项目。放到 `~/832/skills`，再安装或适配给不同工具。

第四，MCP 按权限边界放。

包含密钥、账号、个人授权的 MCP，不要进公开仓库。团队共享的、非敏感的 MCP，才考虑项目级。

第五，别为了统一而统一。

AI 编程工具现在还没有完全统一的配置标准。硬统一目录，最后一定会变成一堆兼容补丁。更稳的做法是：统一原则，尊重各工具的原生入口。

## 9. 结论

我现在不再问“这三个工具能不能共用一个目录”。

更好的问题是：

```text
这是项目级，还是用户级？
这是规则、命令、Skills，还是 MCP？
这是公开可共享，还是只属于我自己？
```

回答清楚这三个问题，目录自然就出来了。

对我来说，最后的分工就是：

```text
832OS/              # 工程规则和项目命令
~/832/skills        # 用户级可复用 Skills
各工具 MCP 配置      # 外部工具连接
```

这套分工不追求表面统一，但它能减少分叉，也更符合真实使用方式。
