## 1. 核心观点

- Claude Code 的真正意义是把模型接入真实世界，不只是写代码更快。
- 工程生产力正在从"人写代码"转向"人设计循环，循环指挥模型"。
- 旧角色边界（工程师、PM、设计师）在消失，留下来的只是能把事情做出来的 builder。
- 组织的预算要从人力堆叠转向自动化复利：
	- 少配人，多给 token。
- **如果品味也会被模型追上，人最后要负责的是价值观和方向感。**

## 2. Claude Code 的起点

- 2024 年底，Anthropic 的 Labs 团队探索下一个产品方向——模型已经能做很多事，但产品形态没接住这些能力。
- 自动补全和问答式 Agent 够不到"替你完成一段工作"。
- 早期 Claude Code 只能完成 10%-20% 的代码，并不神奇。
- **真正的阶跃来自底层模型能力提升**：
	- 从 5 月的 Sonnet 4 / Opus 4 到 11 月的 Opus 4.5。
- 产品层也在补齐：
	- CLI、桌面、移动、Slack、GitHub 应用、Plan mode。
- Anthropic 做这个的核心动机是 AI 安全。
- 编程是模型与现实世界互动最直接的方式
	- ——代码天然有可验证反馈，能编译、能跑、能测试，比生成文本更像一个可验证的世界。

> 编程是模型和现实世界互动的一种方式。

## 3. 工程生产力的变化

- 发布 Claude Code 后，Anthropic 每位工程师代码量增长数倍（"3 倍"数据已过时）。
- 工程团队规模扩大多倍，但生产力没有按传统经验下降
	- ——Claude Code 承担了"问人"和"找上下文"的负担。
- 新人上手从几周缩短到两天。查数据库、了解流程、找工具，直接问 Claude Code。
- 写代码的抽象层一直在上移：打孔卡 → 汇编 → 高级语言 → 今天。Claude Code 只是下一层。
- Boris 的轨迹：IDE → 同时跑 5-10 个 Claude → 卸载 IDE → 写循环来 prompt Claude，循环自己判断该做什么。

## 4. 通才时代

- 传统流程：
	- 研究员访谈 → 设计师出 mock → PM 定范围 → 工程师实现。Claude Code 团队里每个人都在做范围定义、用户交流、设计工程、数据分析和 dashboard。
- 设计师、财务、幕僚长也在发布代码——所有角色在合并成一种 builder。
- 新人可能比资深工程师更自然使用 Claude Code，**没有旧习惯要卸载**。
- 经验的价值在转移：
	- 从"我知道怎么写"到"我知道什么值得做，怎么验收，哪里会出事"。
- 每个新模型都会改变最佳实践，老习惯会变成阻力。
- "技术员工"统一头衔：消除了头衔级别对想法判断的污染。

> 我觉得现在就是通才的黄金时代。想做不止一件事的人，从来没有像现在这么好玩，也从来没有像现在这么容易。

## 5. 组织设计

- 给每个人尽可能多的 token，让他们去实验。
- 项目看起来需要 4 个工程师？放 2 个进去，给足 AI 资源。更少的人力约束逼出自动化。
- 自动化形成复利：
	- 这次做成，下次成本更低。本质是把预算从人力转向 token。
- 公司的产品原则、决策原则不能只写给人看，要变成模型能读取、执行的 skills。
- 对个人同理：`AGENTS.md`、`SKILL.md`、`whoami`、`os/notes` 是把人的判断压进 Agent 工作流的方式。

## 6. 产品取舍

- Claude Code 最早只在终端。
	- 但数据科学家自己装 Node.js、配 API key 来做分析。有人在 Twitter 上用 Claude Code 监控番茄植株和营养供给，Claude 看到花苞时说："这太让人开心了，我们所有的努力都有回报了。"
- 这些信号说明需求已从工程师扩散到主流人群。
- Claude Cowork 八九天完成，100% 用 Claude Code 构建。
- **浏览器方案摩擦太大**：
	- 没法直接访问本地文件、桌面工具和真实工作材料。本机 Agent 仍然重要。

## 7. 品味与价值观

- Boris 曾坚持代码库不能有 class，只能有函数。后来模型大量写 class，业务结果更快，代码质量也没变差——很多自认为的品味，可能只是个人执念。
- 产品品味今天仍是优势，但也会被模型逐步追上。
- 当代码能力、产品想法、用户反馈分析都逐渐被模型覆盖，人类还剩下什么？
	- **持续告诉模型：什么是好结果、什么不能做、什么值得长期坚持、什么事即使能自动化也必须留给人判断。**

> 就像我们教孩子怎样做一个好人一样，我们也会教模型怎样做一个好模型。

## 8. 总结

- **Claude Code 起源**：模型能力领先于产品 → Labs 团队探索 → 从 10% 起步，底层模型跃迁带来阶跃变化
- **工程生产力**：代码量数倍增长，团队扩大但生产力未降，新人两天上手。"写代码"定义上移：IDE → prompt Claude → 写循环
- **团队融合**：传统瀑布流程被打破，所有角色合并为 builder，通才成为团队青睐的类型
- **产品扩展**：非工程师（数据科学家、番茄种植者）主动使用，Claude Cowork 八天用 Claude Code 构建完成
- **组织设计**：预算从人力转向 token，少人力加多 token 逼出自动化
- **人类最后壁垒**：品味优势会被侵蚀，独特价值在于向模型灌输价值观

## 10. 启示

- 问自己：
	- 怎样让 Agent 从一次性执行走向可复用的循环。
- 每个稳定入口都应该问：
	- 有没有减少上下文搬运？有没有形成可复用的判断？有没有把人的原则交给模型？有没有让下一次更便宜？
- `new-note`、`daily-lesson`、`weekly`、`commit-and-push-832-workspace` 本质上在做同一件事：
	- 把重复判断变成模型可执行的工作流。
- 不要把 Agent 看成"帮我写代码的工具"，要看成"我正在训练的一组工作循环"。
