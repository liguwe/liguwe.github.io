## 总结

第 10 课把 system prompt 从`固定大字符串`改成`运行时动态生成`。

- `用户`  告诉 Agent  “这次要做什么”，
- `system prompt` 告诉 Agent  “ 你是谁、你手里有哪些工具、应该在哪个目录工作、还要遵守哪些规则 ” 。
- 它不会直接显示给用户，却会在每次请求时和用户任务一起发给 DeepSeek。
- 但`prompt` 不应该是是写死的，运行时组装, 不硬编码

前几课把这些说明写成一整段`固定文字`还够用。但随着迭代，这段`固定文字`就像一份长期手改的员工手册：
- 项目换了、工具删了、记忆空了，手册里的内容却可能没有同步变化。
- 哪些工具启用、哪些上下文可见、哪些记忆相关、哪些内容必须保持稳定以命中 prompt cache

所以这一课让程序在调用 DeepSeek 前，先看一眼当前环境，然后**动态生成一份 “上岗手册”**。

```text
真实运行状态
-> 整理成 prompt context
-> 选择 identity、tools、workspace、memory
-> 拼成 system prompt
-> 放进 messages
-> 发给 DeepSeek
```

`同一份运行状态`会得到`同一个 context key`。
- key 变化：
	- 重新组装 prompt。
- key 不变：
	- 复用上一次拼好的字符串。

这里先不用记住 `context` 和 `key`。可以暂时把它们理解成：

```text
context = 这次写上岗手册要用到的全部材料
key     = 这些材料的指纹
```

用户入口没有变化：

```bash
hx agent "读取 package.json，只告诉我项目 name"
```

这一课改变的是 **system prompt 的生成方式**。Agent Loop、工具协议、Hooks、上下文压缩和长期记忆继续沿用前面的实现。

## 重点

- `固定大字符串`已经容纳不下不断增加的 Agent 能力。
	- 换项目时，不知道哪些内容该改。
	- 修改一类规则时，容易影响其他规则。
	- 无关内容每轮都发给模型，会**增加 token 和噪音**。
- System prompt 应该来自真实运行状态。
	- 工作目录：来自 `process.cwd()`。
	- 工具：来自真正注册的定义和 handler。
	- Skills： 来自项目技能目录。
	- memory： 来自当前索引文件。
- 每个 section 有明确职责。
	- `identity`：我是谁。
	- `tools`：我能做什么。
	- `workspace`：我在哪里工作。
	- `memory`：项目长期记住了什么。
- 缓存比较的是整份 prompt context。即 给 prompt context 生成一个`唯一指纹`
	- context 一样，旧 prompt 仍然可用。
	- context 变化，必须重新选择 section。
- 两种优化要分开理解。
	- ① 按需加载：
		- 按需加载 section 可以减少发给模型的内容。
	- ② 利用缓存
		- 本地 cache 只减少重复拼字符串，不会取消 DeepSeek 请求。

## 用一条任务贯穿全文

全文只看这一条任务：

```bash
hx agent "读取 package.json，只告诉我项目 name。这个任务只需要一次 read_file，不要使用其他工具。"
```

它会触发两轮模型调用：

- 第一轮
	- HarnessX 收集当前运行状态。
	- 动态组装 system prompt。
	- DeepSeek 返回 `read_file` 工具调用。
- 第二轮
	- HarnessX 再次收集运行状态。
	- Context 没变，命中 prompt cache。
	- DeepSeek 根据工具结果回答 `harnessx`。

### 一个困惑的补充

看到“第二轮 Context 没变”，很容易产生一个疑问：

```text
读取 package.json 后，文件内容已经放进对话了，
Context 怎么可能没变？
```

这里同时出现了两种不同的 Context：

- 对话上下文变了。
	- `messages` 增加了 DeepSeek 发出的 `tool_calls`。
	- `messages` 又增加了 `read_file` 返回的 `package.json` 内容。
- Prompt Context 没变。
	- 工作目录没变。
	- 可用工具没变。
	- Skills 没变。
	- memory 索引没变。

所以第二轮的真实情况是：

```text
对话上下文已经增加 package.json 内容
-> 第二次发给 DeepSeek 的 messages 变了

生成 system prompt 的运行状态没有变化
-> Prompt Context 的指纹没变
-> 复用上一轮已经组装好的 system prompt
```

`[PROMPT] cache hit` 只表示 system prompt 不需要重新拼装，不表示整个请求没有变化。

还要注意：DeepSeek 自己不能直接打开本机的 `package.json`。

- 第一轮，DeepSeek 只是在说：“请 HarnessX 帮我调用 `read_file`。”
	- HarnessX 在本机执行工具，把文件内容放回对话。
- 第二轮，DeepSeek 看到工具结果，才有依据回答项目名。

第二轮再次请求模型前，HarnessX 仍然要准备 system prompt。工作目录、工具、Skills 和 memory 都没变，所以这次直接复用上一轮的上岗手册；新增的文件内容则跟随 `messages` 一起发给 DeepSeek。

终端里最值得观察的是：

```bash
# 第一轮：运行状态生成了一份新 prompt。
[PROMPT] assembled sections=identity,tools,workspace,memory

# DeepSeek 要求读取真实文件。
> read_file path=package.json limit=all

# 第二轮：运行状态没变，复用 prompt。
[PROMPT] cache hit

# DeepSeek 根据工具结果返回答案。
项目 name 是 harnessx。
```

## 为什么固定大字符串已经不够用

早期 system prompt 很短：

```js
const SYSTEM_PROMPT = [
  "你是 HarnessX 编程 Agent。",
  "请使用工具完成用户任务。",
].join("\n");
```

Agent 只有少量能力时，这样写最直接。

到了第 10 课，HarnessX 已经有工具、权限、Hooks、TodoWrite、子代理、Skills、上下文压缩和长期记忆。所有说明继续塞在一段字符串里，会出现三个问题。

### 换项目时，不知道哪些内容该改

项目不同，运行状态也不同：

- 工作目录不同。
- 注册工具不同。
- 项目 Skills 不同。
- memory 索引不同。

固定 prompt 把这些内容写死后，换项目就要人工重写整段文本。

最直接的后果是“手册和现场对不上”：

- 当前程序运行在项目 B，prompt 里还写着项目 A 的目录。
- `read_file` 已经删除，prompt 里还告诉模型可以使用。
- 当前项目没有 memory，prompt 里仍塞着一段空的记忆说明。

模型只能相信收到的内容。Prompt 写错了，它就可能找错目录、调用不存在的工具，或者被无关规则干扰。

运行时组装把变化规则写进代码：

```text
身份规则    -> identity
工具与技能  -> tools
当前目录    -> workspace
长期记忆    -> memory
```

项目变化后，程序读取新状态并重新生成对应 section。

### 修改一处时，容易影响其他规则

一整段字符串没有责任边界。

新增一段 `load_skill` 说明，可能同时碰到工具列表、技能目录、TODO 规则、memory 规则和最终回答要求。

Section 给每类规则一个明确归属：

- 模型误以为某个工具可用，检查 `tools`。
- 模型理解错工作目录，检查 `workspace`。
- 长期偏好注入错误，检查 `memory`。
- 回答身份或风格错误，检查 `identity`。

规则来源清楚后，调试 prompt 才有具体入口。

### 每轮都带全部内容，会增加 Token 和噪音

System prompt 会跟随每次模型请求发送。

当前没有 memory 时，继续发送空 memory 内容只会增加输入 token。无关说明越多，模型也越难聚焦当前任务。

第 10 课先让 `memory` 按真实状态加载：

```js
if (context.memoryIndex) {
  loadedSections.push("memory");
}
```

索引为空时，`memory` 不进入 system message。

以后增加更多可选能力，也可以沿用同一机制：能力真实存在时才加载对应 section。

## 用上岗卡片理解 System Prompt

把 DeepSeek 想成每轮都会重新上岗的程序员。

用户只说：

```bash
读取 package.json，只告诉我项目 name。
```

这句话说明了任务，却没有说明工作环境。HarnessX 还要准备四张上岗卡片：

```text
┌──────────────── identity ────────────────┐
│ 你是 HarnessX 编程 Agent，用中文简短回答 │
└──────────────────────────────────────────┘

┌────────────────── tools ─────────────────┐
│ 可用工具、技能目录、TODO 和 compact 规则 │
└──────────────────────────────────────────┘

┌──────────────── workspace ───────────────┐
│ 当前工作目录是 /Users/liguwe/832/832X    │
└──────────────────────────────────────────┘

┌───────────────── memory ─────────────────┐
│ 当前项目的长期记忆索引                   │
│ 只有索引非空时才放入                     │
└──────────────────────────────────────────┘
```

需要的卡片按固定顺序拼起来，就是本轮 system prompt。

这个类比里：

- 运行时状态是卡片原料。
- Prompt context 是本轮材料清单。
- Section 是各部门提供的卡片。
- System prompt 是装订后的上岗手册。
- Cache 是上岗材料没变时复用旧手册。

这四个英文词可以这样记：

```text
identity  = 工牌：我是谁
tools     = 工具箱：我能做什么
workspace = 工位：我在哪里做
memory    = 便签：以前有哪些长期信息
```

## 三个设计决策

### System Prompt 由真实状态生成

用户输入说明“这次想做什么”。

运行时状态说明“Agent 当前真正能做什么”。

假设用户说：

```bash
帮我读取 README.md。
```

这句话没有出现 `read_file`、`workspace`、`memory` 或 `skill`。HarnessX 仍然要知道：

- `read_file` 当前有没有执行 handler。
- 相对路径从哪个目录开始。
- 项目是否存在 memory 索引。
- 当前有哪些可按需加载的 Skills。

反过来，用户即使要求调用 `deploy`，本地没有注册这个工具，system prompt 也不能把它描述成可用能力。

这就是“真实状态”的含义：不根据用户希望有什么来写，也不根据以前曾经有什么来写，只描述程序这一刻确实拥有的能力。

当前真实来源是：

```text
workspace   <- process.cwd()
tools       <- TOOL_DEFINITIONS 与 TOOL_HANDLERS 的交集
skills      <- SKILL_REGISTRY
memory      <- readMemoryIndex()
```

还有一种看起来很省事的做法：从用户问题里找关键词。

例如，用户提到“记忆”就加载 memory，没提到就不加载。这样会带来一个问题：运行环境明明没有变化，用户只换了一种说法，Agent 收到的能力说明却变了。

```text
同一运行环境
-> 用户换一种表达
-> 加载出不同的系统能力说明
```

因此，当前实现直接读取真实状态。只要现场没变，生成的内容就保持稳定：

```text
同一运行状态
-> 同一 prompt context
-> 同一 system prompt
```

固定 prompt 容易直接查看，但能力变化后容易过期。代码已经删除工具，旧 prompt 仍可能告诉模型该工具存在。

### 每个 Section 都有明确归属

四个 section 也是四个责任边界。

| Section | 内容 | 来源 | 出错时先检查 |
| --- | --- | --- | --- |
| `identity` | Agent 身份、回答方式 | HarnessX 核心规则 | 身份和输出行为 |
| `tools` | 工具、Skills、使用规则 | 工具注册表、技能目录 | 工具是否真实可用 |
| `workspace` | 当前目录、文件边界 | Node.js 进程 | 路径和工作目录 |
| `memory` | 长期记忆索引 | Memory 存储层 | 记忆是否正确注入 |

![238. 技术：HarnessX 第 10 课：让 System Prompt 根据真实状态组装，context 不变就复用缓存 图表 2](/assets/os/mermaid-d576ca5d885a33369c7bb758394ba214af1874e22654d23930f6bdecd78fa017.svg)

随意拼接字符串在内容少时也能运行。Prompt 增长后，规则来源会逐渐失去可追踪性。

具名 section 让错误可以沿着来源反查。

### 确定性 Context Key 决定是否复用

缓存需要回答一个问题：

```text
这次准备 prompt 的材料，和上次完全一样吗？
```

所以 context key 要覆盖整份材料：

```text
workspace + tools + skillCatalog + memoryIndex
```

`stableSerialize()` 会递归排序对象 key。

- 相同状态总能得到相同 key。
- 任意状态变化都会得到新 key。

为什么不能直接随便把对象转成字符串？

因为同一组材料可能只是排列顺序不同：

```js
{ workspace: "/a", memory: "" }
{ memory: "", workspace: "/a" }
```

它们表达的是同一件事。如果字符串结果因为排列顺序不同而变化，程序就会误以为材料变了。

`stableSerialize()` 会先把顺序整理好，再生成字符串。可以把它理解成给整份材料按固定规则做“指纹”：内容相同，指纹就相同；任何真实内容变化，指纹才变化。

每轮无条件重建 prompt 的代码更短，但会把“prompt 到底有没有变化”藏起来。现在用 key 做判断，再通过 `assembled` 和 `cache hit` 日志把结果暴露出来。

![238. 技术：HarnessX 第 10 课：让 System Prompt 根据真实状态组装，context 不变就复用缓存 图表 3](/assets/os/mermaid-09c6e2fc38c145a79ce96f723a7865b225cd705c2696b8c382fef5d6f95a5eb8.svg)

终端日志会直接显示判断结果：

```bash
# Context 产生了变化。
[PROMPT] assembled sections=identity,tools,workspace,memory

# Context 没有变化。
[PROMPT] cache hit
```

这里的 cache 只复用 Node.js 进程里已经拼好的字符串。

- 它不会取消下一轮 DeepSeek 请求。
- 它没有声明命中了 DeepSeek 服务端缓存。
- 真正减少输入 token 的是按需省略无关 section。

大白话讲，cache 省掉的是“重新装订手册”这一步，并没有省掉“把手册寄给 DeepSeek”这一步。

## 四步完成运行时组装

### 收集 Prompt Context

Agent Loop 先收集真正可用的工具：

```js
const tools = TOOL_DEFINITIONS
  .map((definition) => definition.function)

  // 同时有定义和 handler，才是真正可用。
  .filter((tool) => tool && TOOL_HANDLERS[tool.name])

  // Prompt 只需要名称和简介。
  .map((tool) => ({
    name: tool.name,
    description: tool.description,
  }));
```

工具定义像工具目录，handler 像真正能执行工具的机器。两边都有，才算工具真的可用。只有名字却没有 handler，模型即使选择了它，HarnessX 也执行不了。

再把四类材料交给 prompt builder：

```js
const context = buildPromptContext({
  workspace: process.cwd(),
  tools,
  skillCatalog: SKILL_REGISTRY.listDescriptions(),
  memoryIndex: readMemoryIndex(),
});
```

得到的数据大概是：

```json
{
  "workspace": "/Users/liguwe/832/832X",
  "tools": [
    {
      "name": "read_file",
      "description": "读取当前工作目录里的文本文件内容。"
    }
  ],
  "skillCatalog": "- course-blog: ...\n- sync-upstream: ...",
  "memoryIndex": "- [harnessx-explanation-method](...) ..."
}
```

Prompt context 只保存组装 prompt 需要的事实，不执行工具，也不控制 Agent Loop。

它只是备料清单：先把目录、工具、Skills 和 memory 摆到桌面上，下一步再决定哪些内容写进手册。

### 选择本轮 Section

三个 section 始终存在：

```js
const ALWAYS_PROMPT_SECTIONS = [
  "identity",
  "tools",
  "workspace",
];
```

`memory` 根据真实索引决定：

```js
export function assembleSystemPrompt(context) {
  const loadedSections = [...ALWAYS_PROMPT_SECTIONS];

  if (context.memoryIndex) {
    loadedSections.push("memory");
  }

  return {
    prompt: loadedSections
      .map((name) => PROMPT_SECTIONS[name](context))
      .join("\n\n"),
    loadedSections,
  };
}
```

当前四个 section 的职责：

- `identity`
	- Agent 身份和最终回答方式。
- `tools`
	- 真实工具、技能目录和工具使用规则。
- `workspace`
	- 当前工作目录和文件边界。
- `memory`
	- 长期记忆索引，有内容时才加载。

当前最小实现把 Skills 放在 `tools` 中，因为 `load_skill` 本身也是模型工具。

这一步很像从四张卡片里选本轮需要的卡片。身份、工具和工作目录每次都要交代；memory 没有内容时就不放，避免拿一张空卡片占位置。

### 比较 Context Key

`getSystemPrompt()` 负责缓存分支：

```js
export function getSystemPrompt(context) {
  const nextKey = stableSerialize(context);

  if (nextKey === promptCacheKey && promptCacheValue) {
    console.log("[PROMPT] cache hit");
    return promptCacheValue;
  }

  const { prompt, loadedSections } =
    assembleSystemPrompt(context);

  promptCacheKey = nextKey;
  promptCacheValue = prompt;
  console.log(
    `[PROMPT] assembled sections=${loadedSections.join(",")}`,
  );

  return prompt;
}
```

大白话解释：

```text
材料没变 -> 旧手册继续用
材料变了 -> 重新选卡片并装订
```

程序比较的不是用户问题有没有变化，而是“生成 system prompt 的材料”有没有变化。用户在第二轮换了问题，只要工作目录、工具、Skills 和 memory 没变，system prompt 仍然可以复用。

### 更新 System Message

每轮请求 DeepSeek 前，Agent Loop 都会执行：

```js
function updateSystemMessage(messages) {
  const context = buildPromptContext({
    workspace: process.cwd(),
    tools,
    skillCatalog: SKILL_REGISTRY.listDescriptions(),
    memoryIndex: readMemoryIndex(),
  });

  const content = getSystemPrompt(context);
  const systemMessage = messages.find(
    (message) => message.role === "system",
  );

  if (systemMessage) {
    // 后续轮次：替换成当前最新 prompt。
    systemMessage.content = content;
    return;
  }

  // 第一轮：插入 messages 第一项。
  messages.unshift({ role: "system", content });
}
```

第一轮之前：

```json
[
  {
    "role": "user",
    "content": "读取 package.json，只告诉我项目 name"
  }
]
```

更新以后：

```json
[
  {
    "role": "system",
    "content": "运行时组装出来的 system prompt"
  },
  {
    "role": "user",
    "content": "读取 package.json，只告诉我项目 name"
  }
]
```

System prompt 到这里才真正接回 Agent Loop。

可以把 `messages` 想成准备寄给 DeepSeek 的信封。`updateSystemMessage()` 做的事，就是在寄出前把最新上岗手册放到信封最前面；已经有旧手册时，就换成当前版本。

```json
    {
      "role": "user",
      "content": "读取 package.json，只告诉我项目 name"
    },
    {
      "role": "assistant",
      "content": null,
      "tool_calls": [
        {
          "function": {
            "name": "read_file",
            "arguments": "{\"path\":\"package.json\"}"
          }
        }
      ]
    },
    {
      "role": "tool",
      "tool_call_id": "call_xxx",
      "content": "{ \"name\": \"harnessx\", ... }"
    }
  ],
  "stream": false
}
```

DeepSeek 返回：

```json
{
  "choices": [
    {
      "message": {
        "role": "assistant",
        "content": "项目 name 是 harnessx。"
      }
    }
  ]
}
```

`callDeepSeekMessage()` 最终取：

```js
const message = payload?.choices?.[0]?.message;
```

Agent Loop 再打印 `message.content`。

所以完整的一来一回是：

```text
用户给任务
-> HarnessX 带着上岗手册询问 DeepSeek
-> DeepSeek 请求调用 read_file
-> HarnessX 在本机读取文件
-> HarnessX 把结果交回 DeepSeek
-> DeepSeek 根据真实文件内容回答
```

System prompt 负责告诉模型“怎样工作”，工具结果负责告诉模型“刚才实际读到了什么”。两者作用不同。

## 怎么跑

沿用第 0 课的 `npm link` 和 DeepSeek 配置。

执行：

```bash
hx agent "读取 package.json，只告诉我项目 name。这个任务只需要一次 read_file，不要使用其他工具。"
```

关键输出：

```bash
# 第一轮组装 prompt。
[PROMPT] assembled sections=identity,tools,workspace,memory

# 真实工具调用。
> read_file path=package.json limit=all

# 第二轮复用 prompt。
[PROMPT] cache hit

# 最终回答。
项目 name 是 harnessx。
```

验收标准：

- 看见 `[PROMPT] assembled`。
- 看见 `read_file` 真正执行。
- 下一轮看见 `[PROMPT] cache hit`。
- 最终从真实 `package.json` 读出 `harnessx`。

如果 memory 索引为空，第一行会变成：

```bash
[PROMPT] assembled sections=identity,tools,workspace
```

这说明本轮没有加载 `memory`。

## 这一课真正要记住

- Prompt context
	- 组装 system prompt 所需的真实材料清单。
- Prompt section
	- 由一个明确子系统负责的一张提示词卡片。
- `buildPromptContext()`
	- 整理工作目录、工具、技能目录和 memory 索引。
- `assembleSystemPrompt()`
	- 选择需要的 section，并按固定顺序拼接。
- `getSystemPrompt()`
	- 比较 context key，决定复用或重新组装。
- `updateSystemMessage()`
	- 把本轮 prompt 放入 messages。
- 按需加载
	- 省略当前运行状态不需要的 section。
- 进程内 cache
	- 复用本地字符串，不会取消模型请求。

一句话收束：

```text
System prompt 是运行时状态生成的上岗手册：
材料没变就复用，材料变化就重新组装。
```

### 代码细分

`buildPromptContext()` 把输入统一成稳定结构：

```js
export function buildPromptContext({
  workspace,
  tools = [],
  skillCatalog = "",
  memoryIndex = "",
}) {
  return {
    workspace: String(workspace || ""),
    tools: tools.map((tool) => ({
      name: String(tool?.name || ""),
      description: String(tool?.description || ""),
    })),
    skillCatalog: String(skillCatalog || "").trim(),
    memoryIndex: String(memoryIndex || "").trim(),
  };
}
```

`assembleSystemPrompt()` 选择 section：

```js
export function assembleSystemPrompt(context) {
  const loadedSections = [
    "identity",
    "tools",
    "workspace",
  ];

  if (context.memoryIndex) {
    loadedSections.push("memory");
  }

  return {
    prompt: loadedSections
      .map((name) => PROMPT_SECTIONS[name](context))
      .join("\n\n"),
    loadedSections,
  };
}
```

`stableSerialize()` 生成确定性 key：

```js
function stableSerialize(value) {
  if (Array.isArray(value)) {
    return `[${value
      .map((item) => stableSerialize(item))
      .join(",")}]`;
  }

  if (value && typeof value === "object") {
    return `{${Object.keys(value)
      .sort()
      .map((key) =>
        `${JSON.stringify(key)}:${stableSerialize(value[key])}`
      )
      .join(",")}}`;
  }

  return JSON.stringify(value);
}
```

`getSystemPrompt()` 管理 cache：

```js
export function getSystemPrompt(context) {
  const nextKey = stableSerialize(context);

  if (nextKey === promptCacheKey && promptCacheValue) {
    console.log("[PROMPT] cache hit");
    return promptCacheValue;
  }

  const { prompt, loadedSections } =
    assembleSystemPrompt(context);

  promptCacheKey = nextKey;
  promptCacheValue = prompt;
  console.log(
    `[PROMPT] assembled sections=${loadedSections.join(",")}`,
  );

  return prompt;
}
```

`updateSystemMessage()` 接回 Agent Loop：

```js
function updateSystemMessage(messages) {
  // definitions 和 handlers 同时存在，才是真实可用的工具。
  const tools = TOOL_DEFINITIONS
    .map((definition) => definition.function)
    .filter((tool) => tool && TOOL_HANDLERS[tool.name])
    .map((tool) => ({
      name: tool.name,
      description: tool.description,
    }));

  const context = buildPromptContext({
    workspace: process.cwd(),
    tools,
    skillCatalog: SKILL_REGISTRY.listDescriptions(),
    memoryIndex: readMemoryIndex(),
  });

  const content = getSystemPrompt(context);
  const systemMessage = messages.find(
    (message) => message.role === "system",
  );

  if (systemMessage) {
    systemMessage.content = content;
    return;
  }

  messages.unshift({ role: "system", content });
}
```

最后在每轮 DeepSeek 请求前更新：

```js
for (let turn = 1; turn <= AGENT_MAX_TURNS; turn += 1) {
  // 先整理可能过长的 messages。
  await compactBeforeModelCall(messages, stats);

  // 再根据本轮真实状态准备 system prompt。
  updateSystemMessage(messages);

  // 最新 system message 随请求一起发给 DeepSeek。
  const message = await callDeepSeekMessage(messages, {
    tools: TOOL_DEFINITIONS,
    tool_choice: "auto",
  });

  // 后续继续原来的 tool_calls -> 执行 -> role: tool 循环。
}
```

```text
Agent Loop 准备调用 DeepSeek
-> 读取工作目录、真实工具、技能目录、memory 索引

-> context key 变化就重新组装 section
-> 把 prompt 放进 messages
-> 发起 DeepSeek 请求
-> 工具结果回填后进入下一轮
```
