title: "150. Zed编辑器启示：Agent 最大的问题不是不会做，而是让人等"
---

## 1. 重点

- `Zed Agent Panel + claude-code` 这次最有感的地方
	- 是同一套 CLI Agent 能力终于回到了编辑器现场。
	- 而这个 zed 编辑，真的很快，敲字真的很爽
- 人和 Agent 共用工具时，最重要的是让代码、文件、上下文、diff 和对话在**同一个地方**发生。
- `CLI` 做底层能力，`ACP` 做连接协议，编辑器做`人的工作现场`

## 2. 小事

昨天我在本地 Zed 里整理 `Agent Panel` 的 `ACP` 配置，有几个小事记录下
- Zed 仍然是我看代码的地方。
- `claude-code`、`codex-cli`、`cursor-cli`、`gemini-cli` 这些能力可以作为 Agent 接进来。
- `flomo MCP` 这类上下文能力也可以通过 Agent 的配置侧进入工作流。
- 人不需要先离开编辑器，再去终端里重新组织上下文。

最关键的的，Zed 的本地 `sidebar_threads` 里也留下了这条线程：

```text
写 2 篇文章来总结，你需要看看我的线程内容
```

> **所有线程的内容，都是在本地的某个文件，这些东西都是模型的上下文**

## 3. ACP 解决的是`容器问题`

这次 Zed Agent Panel 的体感更像另一个问题：**Agent 应该和人一起待在哪里？**，至少**人的注意力应该在一个地方**

所以 `ACP` 的意义在于让 Agent 能以标准方式嵌进客户端，它解决的是容器问题。

## 4. 边界

- `Codex Desktop` 继续作为强执行线程。
- Zed 继续作为代码阅读和编辑现场。
- Zed Agent Panel 通过 `ACP` 接常用 CLI Agent。
- Cursor 按需作为上下文、MCP 和 Agent 协作补充。
- `general-agent-lab` 继续主读 CLI 源码和最小原型，不因为看到了 ACP 就立刻转向 GUI 大研究。

## 5. 所以

更合理的分工是：

```text
CLI 提供可执行能力
ACP 提供客户端连接方式
编辑器保留人的工作现场
```

## 6. 最后

可能，我还因为 zed 真的很快，启示是，速度一致会被追求，回到Agent，现在最大的问题就是 `等待`
