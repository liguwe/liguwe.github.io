## Branch 和 Worktree，到底啥区别

- **分支是「代码往哪个方向走」**
	- 主干是国道
	- 分支是岔路
	- 修 Bug 走一条岔路，做新功能走另一条岔路，最后汇合。
- **Worktree 是「同时打开几个代码现场」。**
	- 同一个项目，同一个仓库，但在硬盘上铺开成好几个 **独立目录**。
		- 这个目录在改登录，那个目录在修支付，互不干扰。

一句话总结：

> 分支管「路线」，Worktree 管「现场」。

## 什么时候用分支，什么时候，什么时候用 Worktree？结合个人情况来说明

1. **开分支搞**，因为大概率是自己串行的去做而已
	1. 然后 `多个线程`跑不同的任务
2. **同一个分支，开多个 Worktree**
	1. 说实话，很少这么干，因为都是一个人开发，串行着搞
	2. 但是，不这么搞的话，线程间会互相影响，比如 <u>会重复改一个文件，这样会有一定干扰</u>
	3. 并且，这个对你的`任务拆解能力` 有要求
3. **分支 + Worktree 一起上**
	1. 每条分支派一个 Agent 驻场，各配一个 Worktree
4. **全矩阵拉满——Peter 模式**
	1. 多分支、多目录（worktree）、多机器

## 这个时代的独立开发者，应该怎么做

> 大厂的瓶颈从来不在写代码，在流程、在协调、在开会。

### 第一，练的第一项能力不是写代码，是 拆任务。

比如，这个需求怎么拆成五个互不干扰的任务，同时派给五个 Agent

### 第二，基础设施要先于功能投入。

- 测试、CI/CD、代码规范、各类 spec 等等，否则Agent 越多，产出越乱
- **顶级开发者派给 AI 的活儿，大头是<u>维护地基，不是盖楼</u>。**
	- 地基越稳，你敢同时派的 Agent 就越多。**这才是真正的杠杆。**

![20260721_1.webp|702](/assets/os/20260721_1.webp)

### 第三，一个人就是一支队伍，但你要当产品经理，别当码农。

当下，你最稀缺的资源就不再是时间，而是 **<u>判断力</u>**：

- 做什么功能，不做什么功能；
- 哪个任务优先，哪个砍掉；
- Agent 交上来的代码，合还是不合。

> 不缺会写的人， **缺的是 <u>*知道该写什么的人*</u>**

![20260721_feishu-replica-03.webp|718](/assets/os/20260721_feishu-replica-03.webp)

> 工位 → 碳基工位
>
> **分支是排班的、Worktree 是发工位的**

## 最后

- **程序员的角色，从写代码的人，变成了<u>管代码任务的人</u>**
- 打字快？→ 语音输入？
	- **取决于你能同时管理多少个 并行 的 AI 任务、同时协调多少个AI员工。**
	- 所以打字绝对不够，这限制了你的`输入带宽`, 所以，条件允许，疯狂的 `语音输入` 吧
