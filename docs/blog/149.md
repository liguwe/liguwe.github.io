---
title: "149. 关于 CLI：它是 Agent 的底层，但不是人的主界面"
---

## 1. 重点

- `CLI` 适合给 Agent 用，不适合一直给人用。
- 人要的是现场：
	- 代码、文件、上下文、diff、对话都在眼前。
- Agent 要的是入口：
	- 命令、参数、输出、退出码、可脚本化。
- 所以 `CLI` 该留在底层，别把它当人的主界面。

## 2. 小事

我并不是太喜欢 `Claude Code`，核心原因就是它太 `CLI` 了。

我始终觉得 Cli 就是一个过渡产品，终端不是不能用，是不适合承载太多人的现场。

`CLI` 最大的价值是稳定

## 3. CLI 的位置

`CLI` 该待在底层。
- 给 Agent 暴露能力。
- 给脚本串流程。
- 给自动化留入口。
- 给源码阅读留一条最薄的剖面。

读 `Codex CLI`、`opencode`、`gemini-cli`，不是为了以后天天在终端里工作。

是为了看清楚 Agent 系统最基本的骨架：
- 入口怎么进来。
- 工具怎么调用。
- 权限怎么确认。
- 结果怎么返回。

这条线很薄，所以适合学习。

## 4. 人的界面

人的主界面应该更像 Zed、Codex Desktop、Cursor 这种东西。

因为人需要看见现场。

所以 `ACP` 这类东西有意义。

它把 `CLI Agent` 接回编辑器，让 Agent 能用底层入口，人留在更舒服的现场里。

## 5. 判断

**CLI 是 Agent 的底层接口，不是人的主界面。**

以后不要再纠结 “我要不要喜欢 CLI”。

应该问：

```text
这件事是给 Agent 用，还是给人用？
```
